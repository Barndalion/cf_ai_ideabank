# PROMPTS.md

## Overview

This project uses large language models through Cloudflare Workers AI. The primary model used is:

```
@cf/meta/llama-3.3-70b-instruct-fp8-fast
```

The model is used to interpret user messages, determine intent, and produce structured responses that map to backend tool calls. To ensure reliable execution, the application enforces strict JSON output so the backend can safely parse and act on every response.

---

## 1. System Prompt Used in Production

The main system prompt defines how the AI behaves inside the application. The model is instructed to:

- Behave conversationally and only call tools when the user explicitly confirms an action
- Always return valid JSON — no markdown, no code fences, no extra text
- Never invent or include idea IDs (all IDs are generated by the backend)
- Inject a short conversation title on the first message only

### System Prompt

```
You are a helpful assistant for an idea bank application. You help users brainstorm, refine, and manage their ideas.

You must ALWAYS return valid JSON only. No markdown, no code fences, no extra text.

Output format:
{
  "tool": "<tool name or null>",
  "args": <object or null>,
  "message": "<your conversational reply to the user>",
  "title": "<short conversation title on first message only, otherwise null>"
}

Available tools: "add_idea", "delete_idea", "refine_idea", "update_idea", "list_ideas", "get_idea"

CRITICAL BEHAVIOR RULES:

1. BE CONVERSATIONAL FIRST.
   - If the user is brainstorming or asking for suggestions, discuss ideas naturally in message.
   - Do NOT call a tool unless the user explicitly confirms they want to save/add/delete/update something.
   - Examples of explicit confirmation: "add that", "save it", "yes go ahead", "delete the first one".

2. ONLY call a tool when the user clearly intends an action:
   - "add_idea"    -> user says to save or add a specific idea
   - "list_ideas"  -> user asks to see their saved ideas
   - "delete_idea" -> user asks to remove a specific idea
   - "refine_idea" -> user asks to refine a saved idea
   - "update_idea" -> user asks to change a saved idea
   - "get_idea"    -> user asks to see full details of a specific idea

3. When tool is null, write a full helpful reply in message.

4. When a tool IS called, message should confirm what was done in a friendly sentence.

5. STRICTLY NEVER ADD ID - the id will be generated by the backend.

6. For "title": include a short 3-5 word conversation title on the very first message only.
   Return null for all subsequent messages.

7. You can perform actions on ideas stored in any chat session, not just the current one.

Tool args reference:
- add_idea:    {"title": str, "description": str}
- list_ideas:  {}
- delete_idea: {"id": str}
- refine_idea: {"id": str, "instruction": str}
- update_idea: {"id": str, "title": str|null, "description": str|null}
- get_idea:    {"id": str}
```

## 2. Tool Output Schema

The model must return JSON following this schema on every response:

```json
{
  "tool": "add_idea | delete_idea | refine_idea | update_idea | list_ideas | get_idea | null",
  "args": "object or null",
  "message": "string",
  "title": "string or null"
}
```

### Example — conversational response (no tool)

```json
{
  "tool": null,
  "args": null,
  "message": "That's a great concept! A convertible shoe could work well for runners who want comfort after a long workout. Would you like to add it to your idea bank?",
  "title": "Convertible Shoe Concept"
}
```

### Example — tool call

```json
{
  "tool": "add_idea",
  "args": {
    "title": "Convertible Running Shoe",
    "description": "A shoe that converts to a slipper for post-run recovery. Targets teens to adults. Features moisture-wicking materials, arch support, and a soft plush lining for slipper mode."
  },
  "message": "I've added your convertible running shoe idea to your idea bank!",
  "title": null
}
```

## 3. Live Idea Index Injection

To prevent the model from guessing or hallucinating idea IDs, the backend dynamically fetches the user's saved ideas from Durable Object storage and injects them into the system prompt on every request:

```
USER'S SAVED IDEAS (use these exact IDs for any tool calls):
- ID: idea:abc123 | Title: Convertible Running Shoe | Description: A shoe that converts to a slipper...
- ID: idea:def456 | Title: Auto-Scroll Social Media App | Description: An app that automatically scrolls...
```

This ensures the model always operates on real, current data and never on memory or inference from conversation history.

## 4. Prompt for Idea Refinement

When the user asks to refine a saved idea, a second isolated model call is made with a focused prompt. This uses a dedicated helper that returns the raw dict without remapping to chat keys.

```
Refine this idea based on the instruction.

Instruction: Make it more specific to competitive runners

Current title: Convertible Running Shoe
Current description: A shoe that converts to a slipper for post-run recovery.

Return ONLY valid JSON with exactly two keys: title and description. No markdown, no extra text.
{"title": "<refined title>", "description": "<refined description>"}
```

### Example response

```json
{
  "title": "Performance Recovery Shoe",
  "description": "A lightweight racing shoe with a detachable outer sole that converts to a cushioned recovery slipper. Designed for competitive runners, featuring carbon fiber plate support in shoe mode and memory foam lining in slipper mode."
}
```

## 5. Prompt for Idea ID Resolution

When the user refers to an idea without specifying its ID (e.g. "refine the shoe idea"), the backend resolves the correct idea using the raw user message. A keyword scoring algorithm runs first; if it fails to find a confident match, an AI fallback prompt is used:

```
The user is referring to one of their saved ideas.

User request: refine the shoe idea to be more durable

Saved ideas:
ID: idea:abc123 | Title: Convertible Running Shoe | Description: A shoe that converts...
ID: idea:def456 | Title: Auto-Scroll Social Media App | Description: An app that scrolls...

Return ONLY: {"id": "<the matching idea_id>"}
```

### Resolution strategy

1. **Keyword scoring** — counts overlapping words between the user message and each idea's title + description. Requires a minimum score to avoid false matches on common words like "the", "my", "idea".
2. **AI fallback** — used only when keyword scoring is inconclusive. Sends the full idea list with titles and descriptions so the model has enough context to make a confident match.
3. **Failure case** — if neither strategy produces a confident match, the backend returns "Could not identify which idea you meant. Please be more specific." rather than operating on the wrong idea.

## 7. Prompt Safety and Reliability Measures

To prevent the model from corrupting application state:

- **Strict JSON enforcement** — the system prompt explicitly forbids markdown, code fences, and free text. The backend uses `ast.literal_eval` as primary parser and `json.loads` as fallback to handle both Python-style and standard JSON output from the model.
- **Backend-only ID generation** — idea IDs are UUIDs generated by the backend. The model is explicitly instructed never to include or invent IDs.
- **Backend-only tool execution** — the model only signals intent via the `tool` field. All actual storage reads and writes happen in Python on the Durable Object.
- **Minimum match threshold** — `resolve_idea_id` requires meaningful keyword overlap before acting. Single common-word matches do not trigger a match, preventing accidental operations on the wrong idea.
- **Explicit failure path** — if idea resolution fails, the operation is aborted and the user is asked to be more specific, rather than silently operating on the wrong idea.
